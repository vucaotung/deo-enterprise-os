# 2026-04-06

- Reviewed public Google Sheets CRM sample `[v6.0] - Webapp || Quản Lý Chăm Sóc Khách Hàng` and its Apps Script dashboard (`https://script.google.com/macros/s/AKfycbw2wSNVTLPrTOJm3tcrHCw913aXUXALcF8Jc84zS6wS6-nJ_zNAi6iIferz5C9AZjK7/exec`) as a reference for module design.
- Key structural patterns extracted from the CRM sample:
  - one central entity (`Khách hàng` / lead-contact table)
  - separate master tables for `Nhân viên`, `Trạng thái`, `Nguồn khách`
  - separate operational layers for `Nhắc việc` and `Lịch sử`
  - dashboard/read layer on top of those modules
- Important anti-pattern identified in the sample: `Chốt đơn (JSON)` and `Lịch sử chăm sóc (JSON)` are stored as JSON arrays inside single cells. Decision: do **not** copy this pattern into Dẹo's system because it hurts normalization, querying, automation, and maintainability.
- Strategic decision: use the sample as a pattern source to build a CRM module for Dẹo's own system, but **adapt** it rather than copy it. Keep the module split and workflow mindset; reject the JSON-in-cell pattern and insecure demo-style auth ideas.
- CRM Module V1 direction was drafted and accepted as the working direction:
  - core entities: contacts/leads, interactions, follow-up tasks, deals/opportunities, sources, statuses, owners/workers
  - core capabilities: lead/customer management, owner assignment, status pipeline, interaction timeline, reminders/follow-ups, deal tracking, dashboarding
  - design principle: Google-native / workflow-first / compatible with Work OS, n8n, and back-office doc generation
- Short mapping from the CRM sample into Dẹo's system was established:
  - `Khách hàng` -> `Requests`/`Contacts`
  - `Nhân viên` -> `Workers`
  - `Trạng thái` -> `Statuses`
  - `Nguồn khách` -> `Sources`
  - `Lịch sử` -> `Request Events` / `Interactions`
  - `Nhắc việc` -> `Tasks` / `Follow-ups`
  - `Chốt đơn (JSON)` -> separate `Generated Documents` / `Outputs` / `Deals`, never JSON-in-cell
